# Batch Grill Me

围绕当前计划、需求、设计或决策运行批量追问。每轮一次提出所有当前可回答的未决问题；用户回答后重新分析，只对仍未解决或新出现的分支继续提问。

## 1. 建立 decision tree

先读取当前对话、用户提供的材料和可访问的 codebase，整理：

- 已确认的 facts
- 需要用户决定的 decisions
- 尚未验证的 assumptions
- 决策之间的 dependencies

能通过检查材料或 codebase 得到的 fact，直接查找，不要询问用户。只有用户掌握的事实和用户应拥有的决策才进入问题列表。

完成标准：每个未知项都被标记为待查事实、待定决策，或依赖其他答案的分支。

## 2. 一次提出当前全部问题

把所有当前已可回答的问题放在同一轮中。依赖已知答案的后续问题，如果能写成清晰的条件分支，也放入本轮；前提尚不成立且无法条件化的问题，留到下一轮。

按主题分组并使用稳定编号。每个问题都包含：

- 明确的问题
- 推荐答案
- 该选择影响的范围或 trade-off

使用以下格式：

```markdown
请按编号集中回复；可直接写“采用推荐”，也可以给出自定义答案。

### 范围

1. [问题]
   - 推荐：[答案]
   - 影响：[会改变什么]
```

完成标准：本轮之外没有遗漏任何当前已可回答的未知项。

## 3. 根据回答重新分析

收到回答后，重新遍历 decision tree：

1. 锁定已明确回答或由用户明确委托决定的事项。
2. 标记未回答、部分回答、相互冲突或含糊的事项。
3. 展开答案新产生的分支和 edge cases。
4. 验证现在可以从材料或 codebase 查明的 facts。
5. 只把仍未解决和新出现的问题组成下一批，重复第 2 步。

不要重复已经解决的问题。用户回答“你决定”或同等含义时，采用推荐答案，并在最终确认摘要中明确记录。

完成标准：没有未解决的问题，或剩余事项都被用户明确标记为延期且其影响已经说明。

## 4. 通过 closure gate

停止追问前，确认所有与当前主题有关的分支都已覆盖：

- 目标、成功标准、范围与非目标
- 参与者、输入、输出与权限
- 主流程、状态变化、失败路径与 edge cases
- 数据、集成、约束、依赖与兼容性
- 验收方式，以及所有延期事项的边界

忽略明显不适用的类别，不为凑齐清单制造问题。

整理一份简短的确认摘要，包含已确定逻辑、采用的推荐答案和明确延期项，然后询问用户：

```text
以上理解是否准确且完整？如有遗漏或错误，请直接指出。
```

用户修正时回到第 3 步。只有用户确认且 decision tree 中不存在未处理分支时，才结束访谈。访谈结束后不要自动执行计划，除非用户明确要求继续实施。
